# Implementation Plan: MediMind Lab Middleware

**Branch**: `043-lab-middleware` | **Date**: 2026-03-05 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/043-lab-middleware/spec.md`

## Summary

Build a standalone Node.js/TypeScript middleware service that receives lab test results from 10 physical analyzers (via ASTM E1381/E1394, HL7v2 MLLP, and proprietary protocols), converts them to FHIR R4 Observation + DiagnosticReport resources, and sends them to Medplum Cloud. The service runs on a hospital lab PC as a Windows Service.

**Approach**: Test-driven development throughout. Every module gets unit tests first, then integration tests with mock analyzers, then end-to-end tests using the simulators. 100% test coverage target.

## Technical Context

**Language/Version**: TypeScript 5.7+ / Node.js 20+ (ESM modules, strict mode)
**Primary Dependencies**: serialport v12+, express v4, better-sqlite3 v11, @medplum/core v4, winston v3, dotenv v16
**Storage**: SQLite (better-sqlite3) for offline queue + message audit log
**Testing**: Vitest (unit + integration + e2e), mock serial/TCP streams, ASTM/HL7 simulators
**Target Platform**: Windows 10/11 (hospital lab PC), also runs on macOS/Linux for development
**Project Type**: Single Node.js service (no frontend)
**Performance Goals**: 3,000+ results/day (~2 results/min avg), <10 second end-to-end latency per result
**Constraints**: Must run 30+ days without restart, handle internet outages up to 24 hours, auto-reconnect to analyzers within 30 seconds
**Scale/Scope**: 10 concurrent analyzer connections, 1 Medplum Cloud target

## Testing Strategy

### Test Pyramid

```
        /  E2E Tests  \          ← 6 tests: full simulator → FHIR flow
       / Integration    \        ← 4 tests: multi-module pipelines
      /   Unit Tests     \       ← 60+ tests: every function in isolation
     /____________________\
```

### How We Test Without Real Hardware

Real lab analyzers cost $10K-$200K each. We don't need them for testing. Here's how:

1. **Mock Serial Port**: A class that pretends to be a serial port — you can "write" bytes to it and they appear as "received" data. No hardware needed.

2. **Test Fixtures**: Real ASTM/HL7v2 message bytes captured from actual analyzers (from research docs). Stored as text files in `src/simulators/fixtures/`.

3. **Mock Medplum Client**: A fake Medplum client that records what FHIR resources would be created, without actually calling the API. Used by unit tests.

4. **Simulators**: Scripts that act like real analyzers — they connect to the middleware's ports and send realistic data. Used for manual testing and e2e tests.

5. **In-Memory SQLite**: For queue and logger tests, use `:memory:` SQLite database instead of file — fast, isolated, no cleanup needed.

## Key Technical Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| ASTM transport approach | State machine with explicit states (idle/receiving/sending/error) | Matches the E1381 spec exactly, easy to test each state transition |
| HL7v2 parsing | Custom parser (not hl7-standard npm) | We only need ORU^R01 parsing, not a full HL7 engine. Simpler, less dependencies. |
| MLLP framing | Buffer accumulator with VT/FS/CR delimiters | Standard approach, handles partial TCP reads |
| Offline queue | SQLite via better-sqlite3 (synchronous) | Synchronous API means no race conditions. File-based persistence survives crashes. |
| FHIR resource creation | Transaction bundles via MedplumClient.executeBatch() | Atomic — either all resources (Observations + DiagReport) are created, or none are |
| Observation status | Always `preliminary` | EMR handles verification workflow. Middleware just delivers raw results. |
| Medplum auth | OAuth2 client credentials flow | Standard for machine-to-machine. MedplumClient handles token refresh. |
| Auto-reconnect | Exponential backoff: 1s, 2s, 4s, 8s, 16s, max 30s | Prevents hammering a dead connection while recovering quickly |
| Barcode matching | Search Specimen by identifier, then get ServiceRequest via basedOn | Specimen.identifier contains the 8-digit barcode set by MediMind EMR |
| Test runner | Vitest (not Jest) | Already in package.json, faster, ESM-native |
| Mock serial | Custom MockSerialPort extending EventEmitter | Simpler than mocking the serialport library, full control |
